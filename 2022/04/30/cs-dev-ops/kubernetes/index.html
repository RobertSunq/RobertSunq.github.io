kubernetes

大规模容器编排系统

特性

  • 服务发现和负载均衡
    • Kubernetes可以使用DNS名称或者IP地址公开容器，如果进入容器的流量很大，Kubernetes可以负载均和冰粉配网络流量，从而使部署稳定
  • 存储编排
    • Kubernetes允许自动挂在所选择的存储系统。eg. 本地存储、公共云提供商
  • 自动部署和回滚
    • 使用Kubernetes描述已部署的所需状态，可以将受控的速率将实际状态更改为期望状态。比如可以自动化Kubernetes来部署创建新容器，删除现有容器并将它们的所有资源用于新容器
  • 自动完成装箱计算
    • Kubernetes允许指定每个容器所需的CPU和RAM。当容器指定来资源请求时，Kubernetes可以做出更好的决策来管理容器资源。
  • 自我修复
    • Kubernetes重新启动失败的容器、替换容器、杀死不相应用户定义的运行状态检查的容器，并且在准备好服务之前不将其通告给客户端。
  • 密钥和配置管理
    • Kubernetes可以存储和管理敏感信息，例如密码、OAuth令牌和ssh密钥。可以在不重建容器镜像的情况下部署和更新密钥和应用程序配置，也无需在堆栈配置中暴露密钥。

Kubernetes提供了一个可弹性运行分布式系统的框架，可以更轻松管理系统的Canary部署。

平面组件结构

控制平面组件(control plane)

  • kube-apiserver

    API服务器为K8s集群资源操作提供唯一入口，并提供认证、授权、访问控制、API 注册和发现机制。

    Kubernetes API 服务器验证并配置 API 对象的数据， 这些对象包括 pods、services、replicationcontrollers 等。 API 服务器为 REST 操作提供服务，并为集群的共享状态提供前端， 所有其他组件都通过该前端进行交互。

  • etcd

    兼具一致性和高可用性的键值数据库，可以作为保存 Kubernetes 所有集群数据的后台数据库(例如 Pod 的数量、状态、命名空间等）、API 对象和服务发现细节。 在生产级k8s中etcd通常会以集群的方式存在，安全原因，它只能从 API 服务器访问。

  • kube-scheduler

    kube-scheduler 负责监视新创建、未指定运行Node的 Pods，决策出一个让pod运行的节点。

    调度决策考虑的因素包括单个 Pod 和 Pod 集合的资源需求、硬件/软件/策略约束、亲和性和反亲和性规范、数据位置、工作负载间的干扰和最后时限。

  • kube-controller-manager

    k8s在后台运行许多不同的控制器进程，当服务配置发生更改时（例如，替换运行 pod 的镜像，或更改配置 yaml 文件中的参数），控制器会发现更改并开始朝着新的期望状态工作。

    控制器包括:

    • 节点控制器（Node Controller）: 负责在节点出现故障时进行通知和响应
    • 任务控制器（Job controller）: 监测代表一次性任务的 Job 对象，然后创建 Pods 来运行这些任务直至完成
    • 端点控制器（Endpoints Controller）: 填充端点(Endpoints)对象(即加入 Service 与 Pod)
    • 服务帐户和令牌控制器（Service Account & Token Controllers）: 为新的命名空间创建默认帐户和 API 访问令牌

  • cloud-controller-manager

    云控制器管理器使得你可以将你的集群连接到云提供商的 API 之上， 同时可以将云平台交互组件与本地集群中组件分离。

    下面的控制器都包含对云平台驱动的依赖：

    • 节点控制器（Node Controller）: 用于在节点终止响应后检查云提供商以确定节点是否已被删除
    • 路由控制器（Route Controller）: 用于在底层云基础架构中设置路由
    • 服务控制器（Service Controller）: 用于创建、更新和删除云提供商负载均衡器

Node中组件

节点组件在每个节点上运行，维护运行的 Pod 并提供 Kubernetes 运行环境。

  • kubelet

    一个在集群中每个node上运行的代理。 它保证容器都 运行在 Pod 中。kubelet 定期接收新的或修改过的 pod 规范 PodSpecs（主要通过 kube-apiserver）并确保 pod 及容器健康并以所需状态运行。该组件还向 kube-apiserver 报告运行它的主机的健康状况。

    kubelet 不会管理不是由 Kubernetes 创建的容器。

  • kube-proxy

    kube-proxy是集群中每个节点上运行的网络代理， 实现 Kubernetes 服务（Service） 概念的一部分。用于处理单个主机子网划分并向外部世界公开服务。它跨集群中的各种隔离网络将请求转发到正确的 pod/容器。

    kube-proxy 维护节点上的网络规则。这些网络规则允许从集群内部或外部的网络会话与 Pod 进行网络通信。

    如果操作系统提供了数据包过滤层并可用的话，kube-proxy 会通过它来实现网络规则。否则， kube-proxy 仅转发流量本身。

该部分组件说明参考自： https://zhuanlan.zhihu.com/p/409971135

安装kebuadm

  • 一台兼容的Linux主机。Kubernetes项目为基于Debian和Red Hat的Linux发行版以及一些不提供包管理器的发行版提供通用的指令
  • 每台机器2GB或者更多的内存（较少会影响应用的运行内存）
  • 2 CPU核或更多
  • 集群中的所有机器的网络彼此均能相互连接（公网和内网都可以）
    • 设置防火墙放行规则
  • 节点之中不可以用重复的主机名、MAC地址和product_uuid
    • 设置不同的hostname
  • 开启机器上的某些端口
    • 内网通信
  • 禁用交换分区。为了保证kubelet正常工作，必须禁用交换分区
    • 永久关闭

Namespace

命名空间 ： 用来隔离资源，分组。默认只隔离资源，不隔离网络。

命令行操作

# 查询命名空间
[root@qing centos]$ kubectl get namespace
# [root@qing centos]$ kubectl get ns # ns 为namespace缩写

kube-node-lease        Active   6d3h

# 获取部署的应用
[root@qing centos]$ kubectl get pods -A

# -n namespace 查询指定命名空间下面的应用 kubectl get pods -n kubernetes-dashboard

# 创建命名空间
[root@qing centos]$ kubectl create ns kube-qing

[root@qing centos]$ kubectl delete ns kube-qing

YAML操作

apiVersion: v1 # 版本号

  name: kube-qing
EOF

# 应用该文件创建命名空间
[root@qing centos]$ kubectl apply -f ./kube_qing.yaml
# 删除时也可以使用配置文件删除
[root@qing centos]$ kubectl delete -f ./kube_qing.yaml

Pod

运行中的一组容器，Pod是kubernetes中应用的最小单位

下图中为一个pod包含两个容器组成一个小应用

  1. Pod和container都运行在以docker为基础的环境上
  2. container（容器）为dokcer管理的单位，每个应用包装为一个容器在运行
  3. Pod为k8s在container的基础上再包装一层用于作为最小单位用于管理，Pod中可以只包含一个容器也可以包含多个容器

自愈和故障转移

自愈：当Pod出现停机、删除Pod、容器崩溃等时，k8s可以自行重启Pod

故障转移：集群状态下，如果一台机器因为断电等原因掉线，k8s在监控到这种情况后，会将这台机器上之前运行的所有Pod转移到当前集群中状态正常的节点上。

Deployment StatefulSet DaemonSet Job/CronJob

延伸：

除了Deployment之外，k8s还有StatefulSet、DaemonSet、Job等类型资源。统称为工作负载。

  • Deployment：无状态应用部署，例如微服务，提供多副本等功能
  • StatefulSet：有状态应用部署，例如redis，提供稳定的存储、网络等功能
  • DaemonSet：守护型应用部署，例如日志收集组件，在每个机器都运行一份(有且只有一份)
  • Job/CronJob：定时任务部署，例如垃圾清理组件，可以在指定时间运行

有状态应用使用 StatefulSet 部署，无状态应用使用 Deployment 部署

路径重写

官方示例：https://kubernetes.github.io/ingress-nginx/examples/rewrite/

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  annotations:
    ## 重写路径  /$2  部分  即下方(/|$)(.*)捕获的部分都赋予$2
    nginx.ingress.kubernetes.io/rewrite-target: /$2
    
    ## 指定该部分路径是否只可以通过SSL访问
    # nginx.ingress.kubernetes.io/ssl-redirect: true 
    ## 指定在未启用TLS下，也强制重定向到HTTPS
    # nginx.ingress.kubernetes.io/force-ssl-redirect: true
    ## 定义重定向 上下文到 root 即访问rewrite.qing.com将重定向到 rewrite.qing.com/app1
    # nginx.ingress.kubernetes.io/app-root: /app1
    ## 路径上知否使用正则
    # nginx.ingress.kubernetes.io/use-regex: true
  name: rewrite
  namespace: default
spec:
  ingressClassName: nginx
  rules:
  - host: rewrite.qing.com
    http:
      paths:
      - path: /something(/|$)(.*)
        pathType: Prefix
        backend:
          service:
            name: nginx-demo
            port: 
              number: 80

重写后的效果为：

  • rewrite.qing.com/something -> rewrite.qing.com/
  • rewrite.qing.com/something/ -> rewrite.qing.com/
  • rewrite.qing.com/something/new -> rewrite.qing.com/new

网络模型简述

  • 所有的Pod属于Pod层，相互之间是网络共同的，其IP地址的区间为上面设置主节点配置时设置--pod-network-cidr=192.168.0.0/16
  • 根据Pod的功能属性氛围不同的service，service之间的网络也是相关联通的，而且与所有的pod之间网络联通，，其IP地址的区间为上面设置主节点配置时设置--service-cidr=10.96.0.0/16
  • pod之间可以相互直接访问，也可以通过其所属的service访问其他的pod
  • 外部请求流量通常是经由LB之后到达Ingress层做网关流量管控，之后在发送给对应的serivce来调用其具体的pod功能支持

存储抽象

原始的容器挂载目录在宿主机上，但是现在kubernetes在集群环境下，在不同的机器上启动pod，如果其中C机器上的一个pod出现问题退出了，由于故障转移的机制kubernetes可能会在B机器上新建这个出问题的pod，这时pod所挂载的目录就是在两台机器上了，存在物理上的隔离。

PV&PVC

PV：持久卷（Persistent Volume），将应用需要持久化的数据保存到指定位置

PVC

创建PVC

kind: PersistentVolumeClaim # 资源类型 申请书
apiVersion: v1
metadata:
  name: nginx-pvc
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 200Mi
  storageClassName: nfs  # 与PV中穿件的存储空间名字相同

创建Pod绑定PVC

apiVersion: apps/v1
kind: Deployment
metadata:
  labels:
    app: nginx-deploy-pvc
  name: nginx-deploy-pvc
spec:
  replicas: 2
  selector:
    matchLabels:
      app: nginx-deploy-pvc
  template:
    metadata:
      labels:
        app: nginx-deploy-pvc
    spec:
      containers:
      - image: nginx
        name: nginx
        volumeMounts:
        - name: html
          mountPath: /usr/share/nginx/html
      volumes:
        - name: html
         # 不是指定nfs 而是指定PVC去挂载申请到的目录
          persistentVolumeClaim:
            claimName: nginx-pvc

ConfigMap

配置集，抽取应用配置，并且可以自动更新

  1. 现将需要使用的配置 制作成 configMap
  2. 创建pod时饮用配置集
  1. 将配置文件创建为ConfigMap
# 创建文件
[root@qing centos]$ tee ./redis.conf <<-EOF
# rddis 数据需要持久化存储
appendonly yes
EOF

[root@qing centos]$ kubectl create cm redis-conf --from-file=redis.conf
# cm   --> comfigMap 缩写
# --from-file  由什么文件制作配置集

# 查询配置集
[root@qing centos]$ kubectl get cm
NAME               DATA   AGE
redis-conf			   1      8s

# 查询配置集
[root@qing centos]$ kubectl get cm redis-conf -oyaml
# -oyaml 以 yaml形式 输出 redis-conf 配置集

# 下面为该命令的输出
apiVersion: v1
# data配置下 才是真实的数据 key -> 文件名，value -> 配置文件
data:
  # 默认是文件名 
  redis.conf: |  # 下方是配置文件中的内容
    # redis 数据需要持久化存储
    appendonly yes
kind: ConfigMap  # 资源类型为配置集
metadata:
  creationTimestamp: "2022-05-11T11:16:32Z" # 创建时间
  name: redis-conf  # 创建配置的名称
  namespace: default # 命名空间
  resourceVersion: "439924"
  uid: adf679dd-f040-437c-bffd-7f5effb521ec
  1. 创建Pod
# 使用docker 加载自定义的配置启动
[root@qing centos]$ docker run -v /data/redis/redis.conf:/etc/redis/redis.conf \
-v /data/redis/data:/data \
-d --name myredis \
-p 6379:6379 \
redis:latest redis-server /etc/redis/redis.conf

kubernetes yaml文件版

apiVersion: v1
kind: Pod
metadata:
  name: redis
spec:
  containers:
  - name: redis
    image: redis
    command:
      - redis-server
      # redis容器内部的位置 加载的配置文件
      - "/redis-master/redis.conf"  
    ports:
    - containerPort: 6379
    # 卷挂载
    volumeMounts:
    # 容器内的数据目录 /data
    - mountPath: /data
      name: data
    # 容器内的配置目录 挂载方式在下方定义
    - mountPath: /redis-master
      name: config
  # 宿主机中的卷
  volumes:
    # 宿主机的对应配置
    - name: data
      # 随机分配一个空目录
      emptyDir: {}
    
    # 对应上面 名为 config 的配置文件挂载
    - name: config
      # 配置文件是从配置集中获取
      configMap:
        # 要使用的配置集名称
        name: redis-conf
        # 获取配置集中 data的那些项
        items:
        # 引入 redis-conf 配置集中 data 下 key为 redis.conf的配置
        - key: redis.conf
          # 指定上方通过key获取到的配置 放到什么目录下,在容器的/redis-master创建redis.conf
          path: redis.conf
  1. 检查默认配置
[root@qing centos]$ kubectl exec -it redis -- redis-cli

127.0.0.1:6379> CONFIG GET appendonly
127.0.0.1:6379> CONFIG GET requirepass

Secret

Secret 对象类型用来保存敏感信息，例如密码、OAuth 令牌和 SSH 密钥。 将这些信息放在 secret 中比放在 Pod 的定义或者 容器镜像中来说更加安全和灵活。

# kubectl create secret 创建密钥信息
[root@qing centos]$ kubectl create secret docker-registry qing-docker \
--docker-username=qing \
--docker-password=qing123456 \
--docker-email=111111@qq.com
### 格式说明
[root@qing centos]$ kubectl create secret docker-registry regcred \
  --docker-server=<镜像仓库服务器> \
  --docker-username=<用户名> \
  --docker-password=<密码> \
  --docker-email=<邮箱地址>

[root@qing centos]$ kubectl get secret
# 查看对应的yaml文件 可以发现是被加密的
[root@qing centos]$ kubectl get secret qing-docker -oyaml
apiVersion: v1
kind: Pod
metadata:
  name: private-nginx
spec:
  containers:
  - name: private-nginx
    # 该镜像为私有 需要账号密码下载
    image: qing/privatengin:v1.0
  imagePullSecrets:
  - name: qing-docker
