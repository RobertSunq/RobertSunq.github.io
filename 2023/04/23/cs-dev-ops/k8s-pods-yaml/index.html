kubernetes yaml配置详解——Pod篇

原创 kubernetes
cs-dev-ops
发布日期:   2023-04-23
更新日期:   2023-04-23
文章字数:   2.5k
阅读时长:   10 分
阅读次数:  

k8s 集群中对资源管理和资源对象编排部署都可以通过声明样式（YAML）文件来解决，通过使用kubectl 命令直接使用资源清单文件就可以实现对大量的资源对象进行编排部署了。一般情况下，都是通过配置YAML文件来部署集群的。

官网概念文档 : https://kubernetes.io/docs/concepts/

官网示例文档 : https://kubernetes.io/docs/tasks/

官网API文档 : https://kubernetes.io/docs/reference/generated/kubernetes-api/v1.24/

注意：在k8s应用的yaml文件中不支持使用制表符为缩紧，所以要保证所有缩进都是空格。

kubernetes yaml配置详解——Pod篇

运行中的一组容器，Pod是kubernetes中应用的最小单位

  1. Pod和container都运行在以docker为基础的环境上

  2. container（容器）为dokcer管理的单位，每个应用包装为一个容器在运行

  3. Pod为k8s在container的基础上再包装一层用于作为最小单位用于管理，Pod中可以只包含一个容器也可以包含多个容器


apiVersion: v1 # api的版本 需要在命令 `kubectl api-versions` 的查询列表中
kind: Pod # 资源类型 常用 Pod/Deployment/Service/Ingress/ConfigMap/PersistentVolume/PersistentVolumeClaim/Secret
metadata: # 指定资源的元数据配置，对于所有的API对象来说这部分基本相同，是用来表示这个对象资源的，从Kubernetes中找到这个对象的主要依据
  name: qing-pod # 名称，在同一个namespace中需要唯一
  namespace: qing-kube # 命名空间，需要已经存在
  labels: # 设定标签，常用被用于筛选，多组 <key>:<value> 形式
    app: qing-deploy #
    app.kubernetes.io/name: deploy
    app.kubernetes.io/instance: deploy
    app.kubernetes.io/version: 0.0.1
    app.kubernetes.io/managed-by: qing
    app.kubernetes.io/component: controlle
  annotations: # 自定义注解列表，多组 <key>:<value> 形式
    - name: qing-nginx
spec: # 指定资源类型的规格定义  主要存放的是这个对象独有的定义，来描述其做表达的功能
  restartPolicy: Always # 重启策略 默认为：Always 在容器退出后，会立即创建一个相同的容器， 另外还有 OnFailure 和 Never
  nodeSelector: # 节点选择 选择符合该标签需求的节点调度Pod，需要先给主机打上标签  `kubectl label nodes kube-node1 zone=node1`
    zone: node1
  affinity:
    nodeAffinity: # 节点亲和度配置， 可以更加灵活的配置Pod的分配哪一类节点上
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms: # 具体参见 https://kubernetes.io/docs/tasks/configure-pod-container/assign-pods-nodes-using-node-affinity/
          - matchExpressions:
              - key: disktype
                operator: In
                values:
                  - ssd
  initContainers: # 在应用容器启动前（containers中配置的）运行完毕
    - name: install # 参数具体说明 详看 containers
      image: busybox:1.28
      command:
        - wget
        - "-O"
        - "/work-dir/index.html"
        - http://info.cern.ch
  containers: # 容器的配置
    - name: qing-app-container # 容器的名称
      image: nginx:latest # 要使用的镜像，eg. `busybox`, `busybox:1.28`, `10.192.11.14:5000/qing/busybox:latest`
      command: [ 'sh', '-c', 'echo The app is running! && sleep 3600' ] # 启动容器的运行命令，将会覆盖容器的Dockerfile中定义的 ENTRYPOINT
      imagePullPolicy: Never # 每次启动是检查和更新images的策略 Never:每次都不检查 不论本地是否存在, Always: 每次都检查, IfNotPresent: 本地有就不检查 没有则拉取
      args:
        - aaa # 启动容器的命令参数， 对应Dockerfile中CMD参数
        - bbb
      env: # 设置容器的环境变量 通过 env 或 envFrom 字段设置的环境变量将覆盖容器镜像中指定的所有环境变量 环境变量可以互相引用 所以请避免产生循环饮用
        - name: qing # 变量的名称
          value: "/etc/run.sh" # 对应变量的值
        - name: POD_NAME
          valueFrom:
            fieldRef: # 属性参数
              fieldPath: metadata.name # 属性路径 对应元数据
        - name: POD_NAMESPACE
          valueFrom:
            fieldRef: # 属性参数
              fieldPath: metadata.namespace # 属性路径 对应元数据
        - name: SPECIAL_LEVEL_KEY # 从ConfigMap中获取配置
          valueFrom:
            configMapKeyRef: # 从configMap special-config 获取
              name: special-config # ConfigMap 包含你要赋给 SPECIAL_LEVEL_KEY 的值
              key: special.how # 指定与取值相关的键名， 不指定key时获取所有的内容
      resources: # 资源管理  如果只有limits配置没有requests配置 pod的requests配置则默认与 limits相同
        requests: # 容器运行时，需要的最低配置要求 如果所有的节点都不满足该资源请求 Pod 将无限期处于 PENDING 状态
          cpu: 0.1 # CPU资源（核数）eg. 0.1 , 100m   两种表达方式 0.1 = 100m
          memory: 128Mi # 内存使用量
        limits: # 资源限制  注意 如果Pod中所有容器使用的资源超过该限制，则会被关闭，同时如果配置了容器的重启机制，则容器可能会处于kill、restart、kill、restart的循环中
          cpu: 500m # CPU 请求只能使用绝对数量，而不是相对数量。0.1 在单核、双核或 48 核计算机上的 CPU 数量值是一样的
          memory: 1024Mi # 内存单位 基本单位为byte,也是不指定单位时候的默认值 其他单位有 ： E、P、T、G、M、K、Ei、Pi、Ti、Gi、Mi、Ki
          # 除此之外，还可以分派扩展资源，详情看 https://kubernetes.io/docs/tasks/configure-pod-container/extended-resource/
      ports:
        - containerPort: 80 # 容器对外的端口
          name: http-name # 名称
          protocol: TCP # 协议
      # 配置连接页面 : https://kubernetes.io/docs/tasks/configure-pod-container/configure-liveness-readiness-startup-probes/
      livenessProbe: # pod存活的探针    一共有4中方式
        httpGet: # 通过httpGet检查健康度，返回在 200——399之间，认为容器正常
          path: /heath # URL地址
          port: 80 # 检查用端口
          # host: 127.0.0.1 # 主机地址
          scheme: HTTP # 连接主机的方式，另一种为  HTTPS
          httpHeaders: #请求中自定义的 HTTP 头。HTTP 头字段允许重复
            - name: Accept
              value: ""
        # tcpSocket:
        #  port: 8080
        # exec: # 在容器内执行指定命令。如果命令退出时返回码为 0 则认为诊断成功。
        #   command:
        #     - cat
        #     - /tmp/health
        # grpc: # 使用 gRPC 执行一个远程过程调用。 目标应该实现 gRPC健康检查。 如果响应的状态是 "SERVING"，则认为诊断成功。 gRPC 探针是一个 alpha 特性，只有在你启用了 "GRPCContainerProbe" 特性门控时才能使用。
        # tcpSocket: # 对容器的 IP 地址上的指定端口执行 TCP 检查。如果端口打开，则诊断被认为是成功的。 如果远程系统（容器）在打开连接后立即将其关闭，这算作是健康的。
        #   port:
        failureThreshold: 5 # 当探测失败时，Kubernetes 的重试次数。 对存活探测而言，放弃就意味着重新启动容器
        initialDelaySeconds: 10 # 在容器启动10秒后进行第一次检测
        periodSeconds: 10 # 检查间隔时间为 10秒
        successThreshold: 1 # 探测器在失败后，被视为成功的最小连续成功数
        timeoutSeconds: 1 # 检测的超时时间
      readinessProbe: # 就绪探测器  配置与存活探针相同  就绪探测器在容器的整个生命周期中保持运行状态
        failureThreshold: 5
        httpGet:
          path: /healthz
          port: 10254
          scheme: HTTP
        initialDelaySeconds: 10
        periodSeconds: 10
        successThreshold: 1
        timeoutSeconds: 1
      lifecycle: # 生命周期
        postStart: #  启动容器运行之前运行的命令 , k8s在容器创建后立即发送postStart事件，但是对其调用不保证在容器的入口点[entrypoint]之前执行，其与容器代码异步执行，但是容器会一直阻塞等待postStart执行完毕状态才会改变为RUNNING
          exec:
            command:
              - 'sh'
              - 'yum upgrade -y'
        preStop:  # 停止容器前运行的命令, 在容器结束前立即发送 preStop 事件。除非 Pod 宽限期限超时，Kubernetes 的容器管理逻辑 会一直阻塞等待 preStop 处理函数执行完毕
          exec:
            command: ["/bin/sh","-c","nginx -s quit; while killall -0 nginx; do sleep 1; done"]
      volumeMounts: # 挂载持久卷
        - name: nginx-storage1 # 挂载设备的名称，与volumes[*].name 需要对应
          mountPath: /data/nginx/data1 # 容器内部的目录
        - name: nginx-storage2
          mountPath: /data/nginx/data2
        - name: nginx-storage3
          mountPath: /data/nginx/data3
        - mountPath: /redis-master # 容器内的配置目录 挂载方式在下方定义  配置的时使用配置集
          name: config
  volumes: # 定义宿主机上对应的挂载目录    支持多种类型 详见 ：https://kubernetes.io/docs/concepts/storage/volumes/
    - name: nginx-storage1
      emptyDir: {} # 由k8s自动安排一个空目录
    - name: nginx-storage1
      hostPath: #
        path: /data/data2  # 宿主机目录
        type: Directory # 字段可选  其他类型 : DirectoryOrCreate 不存在则创建空目录 权限为 0755, Directory, FileOrCreate 不存在则创建空文件 权限为 0644, File, Socket, CharDevice, BlockDevice .  没有OrCreate的则需要目标提前存在
    - name: nginx-storage3
      nfs: # nfs存储类型挂载 还支持其他类型文件系统
        server: {{节点IP}}
        path: /nfs/data # 节点上的目录
    - name: pvc-storage
      persistentVolumeClaim: # 使用持久卷声明去申请持久卷
        claimName: nginx-pvc
    - name: config
      configMap:  # 配置文件是从配置集中获取
        name: redis-conf # 要使用的配置集名称
        items: # 获取配置集中 data的那些项
          - key: redis.conf # 引入 redis-conf 配置集中 data 下 key为 redis.conf的配置
            path: redis.conf # 指定上方通过key获取到的配置 放到什么目录下,在容器的/redis-master创建redis.conf
  securityContext: # 设置Pod安全上下文 该配置也可以添加到containers https://kubernetes.io/docs/tasks/configure-pod-container/security-context/
    runAsUsr: 100 # 指定 Pod 中的所有容器内的进程都使用用户 ID 100 来运行
    runAsGroup: 3000 # 指定Pod 中所有容器中的进程都以主组 ID 3000 来运行  默认为 root(0)
    fsGroup: 2000 # 卷 data/nginx/data1, data2, data3 及在该卷中创建的任何文件的属主都会是组 ID 2000
  imagePullSecrets: # 如果从需要认证的镜像仓库下载镜像，可以配置Pod访问Secret中创建的密钥，前提是recred已经存在
    - name: recred  # https://kubernetes.io/docs/tasks/configure-pod-container/pull-image-private-registry/
  dnsPolicy: ClusterFirst # Pod的DNS策略， Default/ClusterFirst/ClusterFirstWithHostNet/None  详情查看 ： https://kubernetes.io/docs/concepts/services-networking/dns-pod-service/
  dnsConfig: # 非必填 填写时与dnsPolicy一起使用，dnsPolicy 为 none 时必须指定
    nameservers: # 非必填 Pod的DNS服务器的 IP 地址列表，最多指定3个IP地址，当 nsPolicy 为 none 时 必须包含一个Ip地址
      - 1.2.3.4
    searches: # 非必填 用于在Pod中查找主机名的DNS搜索域的列表
      - ns1.svc.cluster-domain.example
      - my.dns.search.suffix
    options: # 非必填  此属性中的内容将合并到从指定的 DNS 策略生成的选项
      - name: ndots
        value: "2"
      - name: edns0
  terminationGracePeriodSeconds: 30 # 终止活跃探测器或启动探测器的宽限时间
